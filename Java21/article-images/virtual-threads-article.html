Virtual Threads in Java 21:
The End of Thread-Per-Request Nightmares

A practical guide with real benchmarks, diagrams, and production-ready code — based on hands-on learning with Java 21.

📅 March 2026 ⏱ 12 min read ☕ Java 21 · JEP 444

The Problem Every Java Developer Has Hit

You're building a REST API. Every incoming request hits a database, calls an external service, waits for a response. Simple enough. But then traffic spikes — 10,000 concurrent users — and your server starts throwing: 

java.lang.OutOfMemoryError: unable to create new native thread

Sound familiar? This is the wall every Java developer eventually hits when scaling I/O-heavy applications. And for a long time, the only real answer was: write async/reactive code, or buy more hardware.

Java 21 changes that with Virtual Threads — one of the most impactful features added to the language in years, shipped as a permanent feature in JEP 444.

Understanding the Root Cause: Platform Threads

Before we talk about virtual threads, we need to understand why traditional threads hit that wall. A platform thread (what Java has always used) maps 1:1 to an OS thread. Every new Thread() you create corresponds to an actual operating-system thread.

This creates two compounding problems:

① They're Expensive to Create

Each OS thread typically consumes 1–2 MB of stack memory by default. Spin up 10,000 threads and you're looking at 10–20 GB of memory just for stacks — before your application has even processed a single request.

② They Block the OS Thread While Waiting

In an I/O-bound application, a thread spends the vast majority of its life doing nothing — waiting for a DB response, a REST call, a message from Kafka. But the OS thread it owns? Still occupied. Still burning memory and OS scheduling capacity. Just… waiting. 

// Timeline of a typical "handle request" platform thread:

[CPU work: 5ms] → [waiting for DB: 200ms] → [CPU work: 2ms] → [waiting for API: 150ms]
                      ↑                                              ↑
                OS thread blocked,                          OS thread blocked,
                doing nothing                               doing nothing

Thread pools help — but they only shift the problem. With a fixed pool of 200 threads and 10,000 concurrent requests, 9,800 requests queue up waiting for a free thread.

Fig 1 — Platform threads: 3 out of 4 are blocked doing nothing while still occupying an OS thread and memory.

Enter Virtual Threads (Project Loom)

Java 21 ships Virtual Threads (finalized in JEP 444) as part of Project Loom. The core idea is simple but powerful:

A virtual thread is a lightweight thread managed by the JVM, not the OS.

Virtual threads run on top of platform threads — called carrier threads — but they're not permanently tied to one. When a virtual thread blocks (on I/O, sleep, etc.), the JVM unmounts it from its carrier thread. The carrier is then free to run other virtual threads. When the blocked operation completes, the virtual thread is re-mounted on any available carrier and continues.

The result: you can have millions of virtual threads while using only a handful of carrier threads (typically = number of CPU cores). The JVM's scheduler handles the rest.

Fig 2 — 3-layer architecture: millions of VTs (parked or running) on top of a few carrier threads on top of OS threads.

How It Works: Mount & Unmount

The magic is in the mount/unmount lifecycle. A virtual thread only occupies a carrier thread when it's actually executing CPU instructions. The moment it hits a blocking operation — a DB query, a network call, even a Thread.sleep() — the JVM parks it as a lightweight heap object (a few KB) and frees the carrier for other work.

Fig 3 — Mount/unmount lifecycle. The carrier is freed every time the VT parks, and the VT may resume on a completely different carrier thread.

Creating Virtual Threads: 4 Ways

Java 21 gives you four APIs to create virtual threads. Each has its use case.

Method 1 — Thread.startVirtualThread()

The simplest, one-liner approach for quick tasks:

Runnable task = () -> {
    Thread ct = Thread.currentThread();
    System.out.println("Thread: " + ct);
    System.out.println("Is virtual? " + ct.isVirtual()); // true
};

Thread vt = Thread.startVirtualThread(task);
vt.join(); // wait for it to finish

Method 2 — Executors.newVirtualThreadPerTaskExecutor() ★ Recommended

The idiomatic, production-friendly approach. Creates a fresh virtual thread per submitted task:

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {

    List<Future<?>> futures = new ArrayList<>();

    for (int i = 0; i < 10_000; i++) {
        final int idx = i;
        futures.add(executor.submit(() -> processTask(idx)));
    }

    for (Future<?> f : futures) {
        f.get(); // wait for each
    }
} // try-with-resources shuts down executor cleanly

Method 3 — Named Factory with Thread.ofVirtual()

When you need naming for observability and debugging:

ThreadFactory factory = Thread.ofVirtual()
        .name("vt-", 0)   // → vt-0, vt-1, vt-2 ...
        .factory();

try (var exec = Executors.newThreadPerTaskExecutor(factory)) {
    // each task gets a named virtual thread
}

Method 4 — Direct Thread.ofVirtual().start()

Explicit control with optional naming — great for learning and quick experiments:

Thread.ofVirtual()
    .name("vt-", i)
    .start(() -> {
        Thread ct = Thread.currentThread();
        System.out.println(ct + " | isVirtual=" + ct.isVirtual()); // true
    });
💡 Tip: Always use try-with-resources on executors — it calls shutdown() and waits for all tasks to complete automatically.

The Benchmark: Numbers Don't Lie

Here's where things get interesting. Let's compare platform threads vs virtual threads on a realistic mixed workload — some CPU computation (sum of primes) followed by a blocking I/O simulation via Thread.sleep(10ms). 

private static void processTask(int taskId) {
    // ① CPU work: compute sum of primes up to a random limit
    int limit = RANDOM.nextInt(5000) + 1000;
    int primeSum = calculateSumOfPrimes(limit);

    // ② Blocking I/O simulation
    Thread.sleep(IO_SLEEP_MS); // 10ms
}

// Platform threads — fixed pool of CPU cores
try (ExecutorService pool = Executors.newFixedThreadPool(POOL_SIZE)) {
    // submit 10,000 tasks
}

// Virtual threads — one thread per task
try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) {
    // submit 100,000 tasks (10× more!)
}
Fig 4 — Benchmark results. Virtual threads finish 10,000 I/O-bound tasks in under 1 second vs 51 seconds for a platform thread pool.

The math behind why platform threads are slow is straightforward:

Approach Tasks Pool Size Time
Platform Threads 10,000 = CPU cores ~51 seconds
Virtual Threads 10,000 N/A (one per task) < 1 second
Platform Threads 100,000 = CPU cores ~8 minutes
Virtual Threads 100,000 N/A (one per task) ~60 seconds
📐 The math: With 8 CPU cores and 10,000 tasks each sleeping 10ms:
(10,000 tasks ÷ 8 threads) × 10ms ≈ 12,500ms ≈ 12 seconds (best case, no overhead). Virtual threads park during the sleep, so thousands run "concurrently" with only 8 real carrier threads.

3 Things You Must Know Before Using Virtual Threads

Virtual threads are powerful, but they come with three important gotchas that every developer must understand before shipping to production.

Fig 5 — Three critical pitfalls: daemon lifecycle, no pooling, and synchronized pinning.

Pitfall 1 — Virtual Threads Are Daemon Threads

If your main() thread exits before virtual threads finish, they get killed silently. Always ensure the main thread waits: 

// ❌ WRONG — main may exit before VTs are done
for (int i = 0; i < count; i++) {
    Thread.ofVirtual().start(() -> doWork());
}

// ✅ RIGHT — use CountDownLatch to keep main alive
CountDownLatch latch = new CountDownLatch(count);
for (int i = 0; i < count; i++) {
    Thread.ofVirtual().start(() -> {
        try { doWork(); }
        finally { latch.countDown(); }
    });
}
latch.await(); // main waits here until all VTs complete

Pitfall 2 — Do NOT Pool Virtual Threads

This is the most common mistake when adopting virtual threads. The whole point is that they are cheap to create. Putting them into a fixed pool re-introduces the very queuing problem you're trying to solve. 

// ❌ WRONG — don't pool virtual threads
ExecutorService pool = Executors.newFixedThreadPool(200);

// ✅ RIGHT — one virtual thread per task
ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor();

Pitfall 3 — Pinning with synchronized

In Java 21, if a virtual thread enters a synchronized block and then blocks inside it, it becomes pinned — the carrier thread cannot be freed. Use ReentrantLock instead: 

// ❌ RISKY in Java 21 — VT gets pinned if it blocks inside synchronized
synchronized (lock) {
    blockingOperation(); // carrier thread pinned here
}

// ✅ SAFE — VT can unmount freely with ReentrantLock
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    blockingOperation(); // VT parks, carrier is freed
} finally {
    lock.unlock();
}
⚠️ Note: Java 23+ significantly reduces pinning issues with improvements to synchronized. But if you're on Java 21 (the LTS), keep this in mind for high-throughput code paths.

When to Use Virtual Threads (and When NOT To)

Use CaseVirtual Threads?Why
REST API with DB calls ✅ Yes I/O-bound — threads spend most time waiting
Microservice making outbound HTTP calls ✅ Yes High concurrency with lots of blocking waits
Kafka/RabbitMQ consumer workers ✅ Yes Many concurrent tasks with I/O between them
Image/video processing ❌ No CPU-bound — carrier threads are the bottleneck
Heavy JSON serialization ❌ No CPU-bound — no blocking means no benefit
Already using WebFlux/Reactor ⚠️ Maybe You already solved the problem reactively

Real-World Adoption: Spring Boot 3.2+

If you're on Spring Boot 3.2+, enabling virtual threads for every request handler is a single configuration line: 

# application.properties
spring.threads.virtual.enabled=true

That's it. Every incoming HTTP request is handled on a virtual thread. No code changes, no reactive rewrites. Spring takes care of the rest.

🚀 Combined with Spring Data (JDBC), this alone can dramatically improve throughput on database-heavy REST APIs without touching a single line of business logic.

Conclusion

Virtual threads don't make your code "faster" in the sense of executing more CPU instructions per second. What they do is let you run vastly more concurrent tasks without paying the OS thread cost for each one.

The mental model shift is simple:

// Old model: threads are expensive
→ use pools → manage pool sizes → hit bottlenecks → write async code

// New model: threads are cheap
→ one per task → let the JVM manage → scale to millions → keep readable blocking code
If you have an I/O-bound Java application and you're still on Java 8 or 11, upgrading to Java 21 and enabling virtual threads may be the highest-ROI change you make this year.
